==============================================================================
 C2. ENDPOINT DAFTAR NOMOR ADMIN PENANGAN TAKEOVER  (dinamis via RBAC)
==============================================================================
 Owner menentukan (lewat Admin Site) admin mana yang menangani custom order &
 human takeover. Chatbot mengambil daftar nomornya dari sini (bukan env lagi).

 Yang dikerjakan:
   1) app/models/user.py: tambah kolom  handles_takeover Boolean default False.
   2) Endpoint untuk CHATBOT (Depends(require_service_key)):
        GET /admin/takeover-handlers
          Response 200: { "numbers": ["628111...", "628222..."] }
          Isi: kolom nomor_wa_admin dari user (role admin/owner) yang is_active=true
               DAN handles_takeover=true. Boleh >1 (broadcast).
   3) Endpoint untuk OWNER (Depends(require_owner), JWT) untuk men-set kolom itu
      dari Admin Site:
        PATCH /users/{user_id}/takeover-handler
          Request : { "handles_takeover": true }
          Response: { "user_id": 5, "handles_takeover": true }

 Sisi chatbot: sudah memanggil GET /admin/takeover-handlers (kini mock -> []),
   notifikasi SEMUA nomor hasilnya, fallback ke ADMIN_WA_NUMBER bila kosong.
 Selesai jika: Owner set handles_takeover=true pada admin X (yang punya
   nomor_wa_admin) -> GET /admin/takeover-handlers memuat nomor X.


==============================================================================
 C3. FIELD is_available PADA PRODUK  (dipakai chatbot & saat create order)
==============================================================================
 Definisi: produk "tersedia" jika SETIAP bahan pada resepnya (tabel recipes)
 punya stok cukup di stock_items untuk membuat 1 unit. Produk TANPA resep =
 tersedia (is_available=true). Dihitung on-the-fly (tanpa kolom stok di products).

 Yang dikerjakan:
   1) app/services/product_service.py: fungsi is_available(product, db) -> bool
      (cek tiap Recipe.qty_bahan <= stok tersedia di StockItem).
   2) app/schemas/product.py: tambah field di ProductOut:
         is_available: bool
   3) Isi field tsb di response GET /products/ dan GET /products/{id}.
   4) Di POST /orders: tolak item dengan is_available=false -> 400
      { "detail": "product <id> out of stock" }.

 Sisi chatbot: sudah siap konsumsi ProductOut.is_available — get_menu menandai
   item "(sedang tidak tersedia)", add_to_cart menolaknya. (Field absen ->
   dianggap tersedia, aman.)
 Selesai jika: produk yang resepnya kurang bahan -> is_available=false di
   GET /products/, dan POST /orders menolaknya (400).

 C5. ENDPOINT UBAH STATUS ORDER  (pemicu C4 push "ready")
==============================================================================
 Kamu SUDAH bikin order_service.update_order_status (yang sudah POST ke chatbot
 saat status jadi "ready" — itu C4). Tapi belum ada route HTTP yang memanggilnya,
 jadi status order tak bisa diubah & C4 tak pernah ke-trigger.

 Yang dikerjakan:
   - Endpoint (Depends(require_admin_or_owner), JWT — ini aksi Admin/Staff):
       PATCH /orders/{order_id}/status
         Request : { "status": "in_process" }   // pending|in_process|ready|delivered|picked_up
         Response: OrderOut
   - Di dalamnya panggil update_order_status(db, order_id, status). (Fungsi itu
     yang otomatis push ke chatbot saat status="ready".)
   - Pastikan env CHATBOT_URL di backend mengarah ke Chatbot Service
     (mis. http://localhost:8000), bukan default localhost:8001.

 Sisi chatbot: receiver POST /webhook/internal/orders/{id}/ready SUDAH ada &
   teruji — begitu C5 jadi, chain "Admin set ready -> pelanggan dapat WA" jalan.
 Selesai jika: PATCH status order jadi "ready" -> pelanggan otomatis dapat WA
   "pesanan siap".
